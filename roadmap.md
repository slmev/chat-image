# AI 图片生成应用 - 客户端化开发路线图

## 当前状态

项目已完成 Web 应用到 Tauri 桌面客户端的主要客户端化开发，当前进入发布候选验收阶段：

- 已接入 Tauri 2 桌面壳层。
- 已实现 Web/Tauri 双运行时判断。
- 已抽象图片持久化接口。
- Tauri 运行时已支持将生成图片保存到 AppData/images。
- 元数据已从 localStorage 迁移到 Tauri Store。
- 桌面 zip 导入导出、本地图片操作、存储统计和清理、桌面 HTTP 请求已实现。
- 桌面构建链路当前可用，macOS app 和 dmg 已生成。

当前剩余重点是使用真实 API Key 和交互式桌面环境完成发布前人工验收。

## P1: 桌面导入导出

### 4. 桌面导出改为 zip 包

- [x] Web 端继续保留当前 JSON 导出。
- [x] Tauri 端导出 zip 包。
- [x] zip 内包含 `history.json`。
- [x] zip 内包含 `images/` 目录。
- [x] `history.json` 中图片引用使用相对路径，不使用当前机器的 AppData 绝对路径。
- [x] 导出时通过系统保存对话框选择保存位置。

验收标准：

- 导出的 zip 在另一台机器或清空数据后仍可导入。
- zip 内 JSON 不包含大体积 base64。
- zip 内图片数量与历史记录引用一致。

### 5. 桌面导入支持 zip 包

- [x] 读取并校验 `history.json`。
- [x] 将 zip 内图片写入当前机器 AppData/images。
- [x] 重写导入消息中的 `localPath`。
- [x] 支持 replace 和 merge 两种模式。
- [x] 导入失败时不破坏当前数据。

验收标准：

- replace 模式会替换当前历史。
- merge 模式只追加不重复的消息。
- 导入后重启应用，图片仍能正常显示。
- 图片缺失或 zip 损坏时给出明确错误提示。

## 当前推荐开发顺序

功能开发已完成。下一步按 `docs/desktop-release-verification.md` 执行发布前人工验收：

1. 使用真实 API Key 验证配置和测试连接。
2. 生成至少一张图片，验证本地保存、重启恢复和图片预览操作。
3. 验证桌面 zip 导出、replace 导入和 merge 导入。
4. 验证删除消息、删除历史、清空历史和孤儿图片清理。
5. 验收通过后创建发布标签或进入签名、公证、分发流程。

## 技术栈

| 用途 | 方案 |
|------|------|
| 前端框架 | Vue 3 + TypeScript |
| 构建工具 | Vite |
| 状态管理 | Pinia |
| 桌面框架 | Tauri 2 |
| 元数据存储 | Tauri Store |
| 图片文件存储 | AppData/images |
| 文件能力 | Tauri fs/dialog/opener plugins |
| 网络能力 | 浏览器 fetch / Tauri http plugin |
| 测试 | Vitest |
